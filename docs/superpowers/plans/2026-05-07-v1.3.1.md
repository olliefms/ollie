## Architecture rules (non-negotiable)

- `merge_insert` everywhere ‚Äî upserts only, never delete+insert
- RFC3339 UTC+Z on all timestamps
- LanceDB 0.27: use `.only_if()` not `.filter()`; require `ExecutableQuery` and `QueryBase` traits in scope
- Soft deletes ‚Äî status = inactive/cancelled before any hard delete
- Status transitions via action endpoints only, never PUT
- Append-only events table ‚Äî never delete events

## Architecture decisions (from Opus review, 2026-05-07)

### #72 ‚Äî Two-step trips DELETE

- **First DELETE** (trip is active): transition to `Cancelled` ‚Äî current behavior unchanged
- **Second DELETE** (trip already `Cancelled`): hard-delete via `trip_table.delete("id = '...'")` using a new `DbClient::hard_delete_trip(id)` method
- **Events:** do NOT cascade-delete. Orphaned events are acceptable ‚Äî the events table is append-only audit history; deleting events would contradict that contract. Events with a missing `entity_id` are tolerable.
- **Race condition:** LanceDB `delete()` is idempotent; two concurrent hard deletes on the same id both converge safely.
- **OpenAPI + llms.txt:** update `DELETE /api/v1/trips/:id` description to document both 204 cases (soft-cancel if active, hard-delete if already cancelled).
- **Test required:** insert ‚Üí DELETE (soft) ‚Üí GET returns cancelled ‚Üí DELETE (hard) ‚Üí GET returns 404.

### #74 ‚Äî Loads DELETE FK guard

- Check `count_active_trips_for_load(id)` (active = NOT IN `cancelled`, `delivered`, `completed`) before hard delete
- Return `409 Conflict: "load has N active trip(s); cancel or complete them first"` if > 0
- Keep loads as immediate hard delete once clear (no two-step pattern for loads)
- Events: do not block on events ‚Äî same orphan-tolerance as trips
- **OpenAPI + llms.txt:** add `409` response to `DELETE /api/v1/loads/:id`.
- **Test required:** insert load ‚Üí insert active trip referencing it ‚Üí DELETE load ‚Üí 409; then cancel trip ‚Üí DELETE load ‚Üí 204.

### #73 ‚Äî Geocoding address simplification

- Simplification logic lives in `process_facility_geocoding` (pipeline layer), not inside `GeocodingClient`
- Add `simplify_address(s: &str) -> Option<String>` helper; returns `None` if simplified == original (avoid wasted retry)
- Simplification rules (applied in order):
  1. Strip `c/o ...` segments (regex or string split on `c/o`, case-insensitive)
  2. Highway normalization: `US Hwy NN` ‚Üí `US-NN`
  3. Collapse whitespace and trailing punctuation/commas
- If full address returns no Census matches ‚Üí try `simplify_address` ‚Üí retry once
- **Do not** increment `geocode_failure_count` between original and simplified attempts ‚Äî it's one pipeline tick
- On success via simplified address: store Census `matched_address` in `normalized_address`, log `INFO "geocoded {id} via simplified address: {original} -> {simplified}"`
- **Dependency:** check Cargo.toml before adding `regex` crate ‚Äî prefer simple string ops for a patch release
- **Tests:** unit-test `simplify_address` with the two known-failing addresses (pure function, no network); live geocode test stays `#[ignore]`

### #77 ‚Äî Facility contacts in driver stop detail

- Add `DriverFacilityContact { name: String, title: Option<String>, phone: String }` struct in `data.rs` ‚Äî do NOT reuse `models::FacilityContact`; explicit projection keeps the driver API contract stable
- Filter: include only contacts where `phone.is_some()`, project to `{name, title, phone}` ‚Äî drop `email` and `notes` from the driver-facing response
- `facility_opt` is already fetched in the stop detail handler; use `.as_ref()` to extract contacts before `facility_opt` is moved into the address mapping
- Always return `contacts: []` (empty array, never null/missing) so the PWA doesn't need a presence check
- Frontend (`stop-detail.js`): render contacts section below address only if `contacts.length > 0`; each contact: name + title line, `<a href="tel:{phone}">{phone}</a>` link
