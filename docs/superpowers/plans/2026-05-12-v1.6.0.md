## Task 1 — #107: Dummy bcrypt on unknown dispatcher email

**Effort:** Small  
**Risk:** Self-contained  
**Files:** `src/api/dispatcher_portal/auth.rs`

### Problem

`login` handler calls `get_dispatcher_by_email`, gets `None`, immediately returns 401. An attacker can
distinguish "unknown email" from "wrong password" by response timing (no bcrypt work on the unknown-email path).

### Implementation

1. Add a module-level `OnceLock<String>` for a static dummy hash pre-computed at cost 12:

```rust
use std::sync::OnceLock;
static DUMMY_HASH: OnceLock<String> = OnceLock::new();
fn dummy_hash() -> &'static str {
    DUMMY_HASH.get_or_init(|| bcrypt::hash("dummy", 12).expect("bcrypt init"))
}
```

2. In the `login` handler, after `get_dispatcher_by_email` returns `None`, call
   `bcrypt::verify(req.password, dummy_hash())` (ignore the result), then return `Err(AppError::Unauthorized)`.

3. The existing `match dispatcher { None => return Err(Unauthorized) }` becomes:
```rust
let Some(dispatcher) = state.db.get_dispatcher_by_email(&email).await? else {
    let _ = bcrypt::verify(&req.password, dummy_hash());
    return Err(AppError::Unauthorized);
};
```

### Tests

Add to `tests/integration_test.rs`:
```
test_dispatcher_login_unknown_email_returns_401
```
POST `/dispatch/auth/login` with a non-existent email → assert HTTP 401. No timing assertion needed
(timing correctness is inherently hard to test deterministically).

### Commit

```
fix: equalise dispatcher login timing for unknown email (#107)
```

## Task 2 — #109: Batch facility lookup in driver portal `list_trips`

**Effort:** Small  
**Risk:** Self-contained  
**Files:** `src/api/driver_portal/data.rs`

### Problem

`list_trips` resolves `next_stop_name` with one `get_facility_by_id` call per trip (inside `join_all`),
plus two more calls via `resolve_stop_name` (origin/destination). At scale this is O(3N) DB round-trips.

### Implementation

`batch_get_facilities(ids: &[Uuid]) -> Result<HashMap<Uuid, FacilityRecord>, AppError>` already exists
on `DbClient` (introduced for `dispatcher_portal/data.rs` in v1.4.0). Verify the signature before use.

Replace the per-trip facility calls with a single pre-fetch:

1. Before the `join_all` over trips, collect all facility IDs needed:
   - For each trip: origin stop facility_id, destination stop facility_id, next-stop facility_id.
2. Call `state.db.batch_get_facilities(&all_ids).await?` once → `HashMap<Uuid, FacilityRecord>`.
3. Move `resolve_stop_name` and `next_stop_name` resolution to use the pre-fetched map (no async DB call).
4. The closure passed to `join_all` no longer needs `state.db` for facility lookups — only for
   truck/trailer unit resolution, which remains per-trip.

```rust
// Before join_all: collect all facility IDs
let fac_ids: Vec<Uuid> = filtered.iter().flat_map(|trip| {
    let mut ids = vec![];
    if let Some(s) = trip.stops.first()  { if let Some(id) = s.facility_id { ids.push(id); } }
    if let Some(s) = trip.stops.last()   { if let Some(id) = s.facility_id { ids.push(id); } }
    if let Some(s) = trip.stops.iter().find(|s| s.actual_arrive.is_none()) {
        if let Some(id) = s.facility_id { ids.push(id); }
    }
    ids
}).collect::<std::collections::HashSet<_>>().into_iter().collect();
let fac_map = state.db.batch_get_facilities(&fac_ids).await.unwrap_or_default();
```

Then pass `fac_map` (via `Arc` or clone into each async closure) for name resolution.

### Tests

No new test. Existing trip-list tests cover correctness. Add a brief comment in the code explaining
the batch approach.

### Commit

```
perf: batch facility lookups in driver portal list_trips (#109)
```

## Task 3 — #115: Relax `assign_trip` status check (Option B)

**Effort:** Medium  
**Risk:** Low-medium (state machine logic change; no schema migration)  
**Files:** `src/api/trip_actions.rs`, `src/api/dispatcher_portal/data.rs`, `tests/integration_test.rs`

### Problem

`assign_trip` rejects with 409 when driver/truck status is `Assigned` or `Dispatched`, blocking
pre-assignment of future trips while a driver is on an active load.

### State machine note

`update_driver_status`, `update_truck_status`, `update_trailer_status` in the DB layer do a direct
fetch-modify-upsert and do NOT call `can_transition_to`. The `can_transition_to` check is only enforced
by the admin PUT endpoint for user-initiated status changes. Action endpoints (assign, dispatch, complete)
call the DB update functions directly and can set any valid status value. This means "no-op"
self-transitions (e.g. `Assigned → Assigned`) are safe to call — they just write the same value.

### Design (Option B — no schema migration)

**assign_trip changes (admin `trip_actions.rs` + dispatcher `data.rs`):**

- Driver check: allow `Available`, `Assigned`, `Dispatched`. Explicitly reject `Inactive`:
  ```rust
  if matches!(driver.status, DriverStatus::Inactive) {
      return Err(AppError::Conflict(format!("driver {} is not available for assignment", body.driver_id)));
  }
  ```
- Truck check: allow `Available`, `Assigned`, `Dispatched`. Explicitly reject `OutOfService` and `Inactive`:
  ```rust
  if matches!(truck.status, TruckStatus::OutOfService | TruckStatus::Inactive) {
      return Err(AppError::Conflict(format!("truck {} is not available for assignment", body.truck_id)));
  }
  ```
- Trailer check: allow `Available`, `Assigned`. Reject `Dispatched`, `OutOfService`, `Inactive`:
  ```rust
  if !matches!(trailer.status, TrailerStatus::Available | TrailerStatus::Assigned) {
      return Err(AppError::Conflict(format!("trailer {} is not available for assignment", trailer_id)));
  }
  ```
- Status update: only call `update_driver_status(driver_id, Assigned)` when driver is currently
  `Available`. If `Assigned` or `Dispatched`, skip the status update call (driver is already busy on
  another trip; do not downgrade their status). Safe because `update_driver_status` does not check
  `can_transition_to`.
- Same skip logic for truck: only set to `Assigned` if currently `Available`.
- For trailers: only set to `Assigned` if currently `Available`; skip if already `Assigned`.

```rust
// driver status update (replace unconditional call at trip_actions.rs:90)
if driver.status == DriverStatus::Available {
    state.db.update_driver_status(body.driver_id, DriverStatus::Assigned).await?;
}
// truck status update (replace unconditional call at trip_actions.rs:91)
if truck.status == TruckStatus::Available {
    state.db.update_truck_status(body.truck_id, TruckStatus::Assigned).await?;
}
// trailer loop (replace trip_actions.rs:92-94)
for &trailer_id in &body.trailer_ids {
    let trailer = state.db.get_trailer_by_id(trailer_id).await?;
    if trailer.status == TrailerStatus::Available {
        state.db.update_trailer_status(trailer_id, TrailerStatus::Assigned).await?;
    }
}
```

**Note:** The trailer fetch above is a second DB call (first was for the availability check loop above).
Combine the two trailer loops into one — fetch once, check eligibility, conditionally update status.

**dispatch_trip changes (ADMIN `trip_actions.rs` only):**

The dispatcher portal has NO `dispatch_trip` handler (only `assign_trip` and `unassign_trip` at
`src/api/dispatcher_portal/mod.rs:23-24`). Apply this guard only to `src/api/trip_actions.rs`.

Add guard after the existing `TripStatus::Assigned` check:
```rust
if let Some(driver_id) = existing.driver_id {
    let driver = state.db.get_driver_by_id(driver_id).await?;
    if driver.status == DriverStatus::Dispatched {
        return Err(AppError::Conflict(
            "driver is already dispatched on another trip".into()
        ));
    }
}
if let Some(truck_id) = existing.truck_id {
    let truck = state.db.get_truck_by_id(truck_id).await?;
    if truck.status == TruckStatus::Dispatched {
        return Err(AppError::Conflict(
            "truck is already dispatched on another trip".into()
        ));
    }
}
```

**OpenAPI:** Update 409 description for `assign_trip` in BOTH `trip_actions.rs` and
`dispatcher_portal/data.rs` to:
`"Conflict — driver/truck/trailer not eligible for assignment (inactive/out-of-service) or invalid status transition"`.

### Status update logic summary

| Driver current status | assign_trip update | dispatch_trip |
|---|---|---|
| Available | → Assigned | → Dispatched |
| Assigned | no change (skip) | → Dispatched |
| Dispatched | no change (skip) | reject 409 |
| Inactive | reject 409 | n/a |

### Dispatcher portal `assign_trip`

`src/api/dispatcher_portal/data.rs` has its own `assign_trip` function (lines 312–). Apply identical
eligibility-check and skip-if-busy status update logic. No `dispatch_trip` changes needed there.

### Tests

Add to `tests/integration_test.rs`:

1. `test_assign_trip_when_driver_is_dispatched_succeeds`
   - Create driver, truck, two trips.
   - Assign + dispatch trip A → driver becomes `Dispatched`.
   - Assign trip B to same driver → expect 200.
   - Assert trip B has `driver_id` set; driver status remains `Dispatched`.

2. `test_dispatch_trip_when_driver_already_dispatched_fails`
   - Create driver, truck, two trips.
   - Assign + dispatch trip A → driver becomes `Dispatched`.
   - Assign trip B to same driver → 200.
   - Attempt to dispatch trip B → expect 409.

## Task 4 — #117: CSS duplicate `.passkey-status` declarations

**Effort:** Tiny  
**Risk:** None  
**Files:** `static/driver/css/components.css` (lines 664–674)

### Problem

`.passkey-status--success` duplicates `margin-top: 0.5rem` and `font-size: 0.875rem` from the base
`.passkey-status` rule. In BEM, both classes are applied together (`class="passkey-status passkey-status--success"`),

### Fix

Remove `margin-top` and `font-size` from `.passkey-status--success`, keeping only its unique properties
(`color`, `font-weight`):

```css
.passkey-status--success {
  color: var(--color-success);
  font-weight: 600;
}
```

After edit, verify no raw hex color values were introduced. Inspect the changed lines visually; the passkey-status rules contain no color values after the fix (modifier only keeps `color: var(--color-success)` and `font-weight: 600`). For the broader hex scan after all CSS tasks, use: `grep -Pn '#[0-9a-fA-F]{3,8}\b' static/driver/css/components.css` — should return only the `:root` token definitions in `base.css`, not any component-level hex.
