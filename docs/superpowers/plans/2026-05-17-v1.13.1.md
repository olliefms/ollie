**Investigation hypothesis:** `has_prev = anchors.iter().any(|(a, _)| *a < lo)` (line 308) is computed against the *current* page's `lo`. When the user clicks "previous week", the API returns a new `has_prev` based on the new `lo`. If the user has trips spanning multiple past weeks, `has_prev` should remain true while older trips exist. If it's flipping false too early, the bug is likely:

- (a) `anchors` only contains trips on the *current visible* page (it doesn't — it's built from `past_only`, which iterates all_trips), OR
- (b) `bound_week` rounds an anchor to its Sunday correctly, but `lo` is the requested week's Sunday in UTC — comparing UTC anchors to a UTC `lo` correctly bins trips, but `has_prev` requires `a < lo`, and an anchor that falls exactly on `lo` (a delivery at Sunday 00:00 local) gets bucketed in the *current* week (`a >= lo && a < hi`) but also passes `a >= lo`, so has_prev correctly excludes it. Likely fine.
- (c) The frontend stepper disables prev when `week.has_prev` is false but never re-fetches the new `week.has_prev` after navigation. Check `trips-past.js:67` — `onChange: load` re-calls `load(weekStart)` which calls `apiFetch('/trips?tab=past&week_start=' + weekStart)` and sets `weekInfo = data.week` (line 40). That part looks fine.
- (d) **Most likely:** `earliest_week_start` is computed correctly but `latest_week_start` returns the bound of the most recent trip, which may be in the past. The date input `max` is `latest_week_start`. If today's "this week" is *after* `latest_week_start`, the date picker disallows jumping to "this week". Verify this doesn't break the prev-button flow.
- (e) Possible real bug: when computing `has_prev`, the comparison `*a < lo` is correct, but `anchors` is sorted ascending and `anchors.first()` returns the earliest trip. If the user's earliest trip is exactly at the boundary, off-by-one. Check the test data carefully.

## Task 6: #188 — Escape `"` and `\` in `sanitize_filename`

**Files:**
- Modify: `src/api/utils.rs`
- Test: extend `src/api/utils.rs::tests`

Header injection via `\r`/`\n` is already blocked. A filename containing `"` (the quote that delimits the value in `Content-Disposition: attachment; filename="..."`) would break value parsing — worst case loss of the filename, not exploitable, but should be hardened anyway.

The HTTP spec way is to backslash-escape `"` and `\` inside the quoted-string. We'll do that.

```rust
    }

    #[test]
```

- [ ] **Step 3: Implement**

Replace the body of `sanitize_filename` in `src/api/utils.rs`:

```rust
pub fn sanitize_filename(name: &str) -> String {
    // Strip CR/LF first (header injection), then backslash-escape `\` and `"`
    // because the value is interpolated into a quoted Content-Disposition
    // header: `filename="..."`. Backslash must be replaced first so the
    // backslashes inserted for `"` aren't re-escaped.
    let stripped: String = name.chars().filter(|c| *c != '\r' && *c != '\n').collect();
    stripped.replace('\\', r"\\").replace('"', r#"\""#)
}
```

```rust
        TripStatus::Dispatched | TripStatus::InTransit => TripTab::Current,
        TripStatus::Assigned => {
            let first_stop = trip.stops.first();
            let parsed = first_stop
                .and_then(|s| {
                    s.scheduled_arrive
                        .as_deref()
                        .and_then(|sa| parse_stop_time(sa, s.timezone.as_deref()))
                });
            match parsed {
                Some(dt) if dt <= chrono::Utc::now() => TripTab::Current,
                // Safe default per AGENTS.md: pre-departure trips go to Upcoming.
                _ => TripTab::Upcoming,
            }
        }
        TripStatus::Planned => TripTab::Upcoming,
    }
}
```
