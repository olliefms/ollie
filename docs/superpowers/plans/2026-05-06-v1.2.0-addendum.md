## Resolved Decisions

### 1. RFC3339 UTC+Z Mandate
All timestamps in the `events` table (and all new tables) must use RFC3339 with UTC `Z` suffix and fixed precision (e.g. `2026-05-06T14:30:00.000Z`). This ensures lexicographic ordering works correctly for range queries. Enforce in `append_event()` signature and validate at API boundaries.

### 2. Scaffold PR (lands before any parallel work)
A single scaffold PR must land on `v1.2.0` before Wave 1 begins. It creates empty module files and registers placeholders in shared registry files to eliminate merge conflicts across parallel sessions. The scaffold PR contains:

- `src/events/mod.rs` ŌĆö stub hooks only
- `src/models/{driver,truck,trailer,trip,event}.rs` ŌĆö empty, pub mod declared
- `src/api/{drivers,trucks,trailers,trips,trip_actions,events}.rs` ŌĆö empty routers
- `src/db/{driver_ops,truck_ops,trailer_ops,trip_ops,event_ops}.rs` ŌĆö empty impl blocks
- `src/models/mod.rs` ŌĆö new modules declared
- `src/db/mod.rs` ŌĆö `DbClient` fields and `new()` updated with placeholder table handles
- `src/api/mod.rs` ŌĆö `ApiDoc::paths(...)` and `schemas(...)` updated with placeholder entries; router wired to empty handlers

### 3. Merge-Insert Mandate
Issue #19 (non-atomic delete+insert) must land first in Wave 1. All new tables in Waves 2, 3, and 4 must use `merge_insert` from day one. Do not copy the v1.1 delete+insert pattern. This applies to every status transition and update operation on drivers, trucks, trailers, trips, and events.

### 4. Trip Number Generation
Trips use the same generation pattern as loads: `T-YYYY-NNNN` (e.g. `T-2026-0001`). Implement `next_trip_number(year)` in `db/trip_ops.rs` following `next_load_number(year)`.

### 5. Two Separate Stop Structs
`LoadStop` and `TripStop` are separate structs. Do not share a single struct.

**LoadStop** (extends existing `Stop` in `src/models/load.rs` with new fields from #27):
- Adds: `scheduled_arrive_end`, `actual_arrive`, `actual_depart`, `expected_dwell_minutes`, `detention_free_minutes`, `detention_grace_minutes`
- Does NOT have: `load_stop_index`, `name` (freeform), or trip-specific stop types

**TripStop** (new struct in `src/models/trip.rs`):
```rust
pub struct TripStop {
    pub sequence: u32,
    pub stop_type: TripStopType, // origin|fuel|pickup|delivery|relay|empty_move|maintenance|terminal
    pub facility_id: Option<Uuid>,
    pub name: Option<String>,       // freeform, for stops without a facility record
    pub load_stop_index: Option<u32>, // links to load.stops[n] for pickup/delivery stops
    pub scheduled_arrive: Option<String>,
    pub scheduled_arrive_end: Option<String>,
    pub actual_arrive: Option<String>,
    pub actual_depart: Option<String>,
    pub expected_dwell_minutes: Option<u32>,
    pub detention_free_minutes: Option<u32>,
    pub detention_grace_minutes: Option<u32>,
    pub notes: Option<String>,
}
```

### 6. Detention Defaults
Hardcoded in calculation logic only. No facility-level or trip-level default fields. When `detention_free_minutes` is absent, use 120. When `detention_grace_minutes` is absent, use 15. Calculation is derived at invoice time, never stored.

Early arrival (driver arrives before `scheduled_arrive`) is treated as on-time ŌĆö grace period check is `actual_arrive - scheduled_arrive > grace_minutes`, which is negative for early arrivals and therefore always passes.

### 7. Embeddings on All New Entities
All new tables get vector embeddings. At Ollama's cost (milliseconds + minimal storage), semantic search should be available on everything. Use the same `OLLAMA_EMBED_DIM` (default 768) as blobs and loads.

Entities and what to embed:
- **Driver**: embed `name + notes`
- **Truck**: embed `unit_number + make + model + notes`
- **Trailer**: embed `unit_number + owner_name + trailer_type + notes`
- **Trip**: embed `trip_number + stop names/addresses + notes`
- **Event**: embed `event_type + payload + actor`

Create IVF_PQ vector indices following the existing pattern in `db/mod.rs`.

### 8. Cascade Atomicity Policy
Cross-table writes (trip + load + events) are best-effort. If any write fails, return 500. Do not attempt rollback or compensating writes. Log the failure with full context (which table failed, what succeeded). The dispatcher retries. This is acceptable because valid data and a healthy LanceDB instance will essentially never produce a partial failure.

### 9. Load Cancellation ŌĆö Trips Must Be Resolved First
`DELETE /loads/:id` (cancel) returns **409 Conflict** if any trips for that load have status `planned`, `assigned`, `dispatched`, or `in_transit`. The dispatcher must resolve all active trips before cancelling the load. This requires deliberate human reasoning about repositioning, deadhead costs, etc. ŌĆö the backend does not auto-cascade.

### 10. Multi-Trip Relay ŌĆö Cancelled Trip Does Not Deliver Load
If trip 2 of a relay is cancelled after trip 1 is delivered, the load remains `in_transit`. It is not considered delivered. A new trip must be created to complete the load. This requires immediate dispatcher attention.

### 11. Trip Status Enum and Transition Matrix

```
planned ŌåÆ assigned ŌåÆ dispatched ŌåÆ in_transit ŌåÆ delivered
assigned ŌåÉ dispatched          (undispatch, only before in_transit)
planned ŌåÆ cancelled
assigned ŌåÆ cancelled
dispatched ŌåÆ cancelled         (only before in_transit)
in_transit ŌåÆ [no cancel]       (must relay ŌĆö create a new trip)
delivered ŌåÆ [terminal]
```

Full `can_transition_to` matrix:

| Current | Target | Via |
|---------|--------|-----|
| `planned` | `assigned` | `/trips/:id/assign` |
| `assigned` | `planned` | `/trips/:id/unassign` |
| `assigned` | `dispatched` | `/trips/:id/dispatch` |
| `dispatched` | `assigned` | `/trips/:id/undispatch` |
| `dispatched` | `in_transit` | first pickup stop departs (automatic) |
| `in_transit` | `delivered` | final stop departs (automatic) |
| `planned` | `cancelled` | `/trips/:id/cancel` |
| `assigned` | `cancelled` | `/trips/:id/cancel` |
| `dispatched` | `cancelled` | `/trips/:id/cancel` |

`undispatch` cascades driver/truck/trailers back to `assigned` (not `available` ŌĆö they remain assigned to the trip).

### 12. Full Load Ōåö Trip Cascade Matrix

Load status transitions driven by trip events:

| Trip event | Precondition | Load transition |
|-----------|-------------|----------------|
| Trip 1 assigned | Load is `planned` | Load ŌåÆ `assigned` |
| Trip 1 dispatched | Load is `assigned` | Load ŌåÆ `dispatched` |
| Trip 1 undispatched | No other dispatched trips | Load ŌåÆ `assigned` |
| First pickup stop departs | Load is `dispatched` | Load ŌåÆ `in_transit` |
| All trips `delivered` | Load is `in_transit` | Load ŌåÆ `delivered` |
| Last active trip cancelled | Load is `planned` or `assigned` | Load ŌåÆ `planned` |
| Trip cancelled while load `dispatched` | Other trips still active | No load change |
| Trip cancelled while load `in_transit` | ŌĆö | Load stays `in_transit` (relay required) |

Load transitions that remain explicit dispatcher actions (NOT driven by trips):
- `delivered` ŌåÆ `invoiced`: `POST /loads/:id/invoice`
- `invoiced` ŌåÆ `settled`: `POST /loads/:id/settle`
- Any ŌåÆ `cancelled`: `POST /loads/:id/cancel` (409 if active trips exist)

### 13. Removed Load Action Endpoints
The following endpoints are **removed** in v1.2 ŌĆö their state transitions are now driven by trip events:

- `POST /loads/:id/assign`
- `POST /loads/:id/dispatch`
- `POST /loads/:id/in_transit`
- `POST /loads/:id/deliver`

Remaining load action endpoints (kept):
- `POST /loads/:id/invoice`
- `POST /loads/:id/settle`
- `POST /loads/:id/cancel` (with 409 guard)

### 14. Action Endpoint Payload Shapes

**`POST /trips/:id/arrive` (stop arrive):**
```json
{ "seq": 0, "actual_arrive": "2026-05-10T08:15:00Z" }
```

**`POST /trips/:id/depart` (stop depart):**
```json
{ "seq": 0, "actual_depart": "2026-05-10T10:30:00Z" }
```

**`POST /trips/:id/late`:**
```json
{ "seq": 0, "eta": "2026-05-10T09:30:00Z", "notes": "traffic on I-80" }
```

**`POST /trips/:id/check-call`:**
```json
{ "location": "Laramie, WY", "notes": "fueling up", "eta_next_stop": "2026-05-10T14:00:00Z" }
```

**`POST /trips/:id/assign`:**
```json
{ "driver_id": "uuid", "truck_id": "uuid", "trailer_ids": ["uuid"] }
```
Returns 409 if driver, truck, or any trailer is not `available`.

**`POST /trips/:id/undispatch`:**
No body. Returns 409 if trip is `in_transit` or beyond.

### 15. Geocode Permanent Failure
Issue #20 (failed geocodes never retried) must also handle the permanent failure case. After 3 consecutive failed geocode attempts, set a `geocode_status = "permanently_failed"` field on the facility record and stop requeueing. Expose `geocode_status` in `FacilityResponse` so dispatchers can identify and manually correct bad addresses.

## Model Field Reference

### Driver
```
id, trip_number*, name, phone?, email?, license_number?, license_state?,
license_expiry?, status (DriverStatus), notes?, embedding,
owner_id, created_at, updated_at
```
*drivers don't have a trip_number ŌĆö `id` is sufficient.

### Truck
```
id, unit_number, year?, make?, model?, vin?, plate?, plate_state?,
status (TruckStatus), notes?, embedding, owner_id, created_at, updated_at
```

### Trailer
```
id, unit_number, owner (TrailerOwner: fleet|carrier|customer|other),
owner_name?, year?, make?, trailer_type?, length_ft?, vin?, plate?,
plate_state?, status (TrailerStatus), notes?, embedding,
owner_id, created_at, updated_at
```
Validation: `owner != fleet` requires `owner_name`.

### Trip
```
id, trip_number (T-YYYY-NNNN), load_id?, sequence,
driver_id?, truck_id?, trailer_ids (Vec<Uuid>),
status (TripStatus), stops (Vec<TripStop>), notes?,
embedding, owner_id, created_at, updated_at
```
`load_id` is nullable ŌĆö trips can exist without a load (empty moves, repositioning, maintenance).

### Event
```
id, entity_type, entity_id, event_type, payload (JSON string)?,
actor?, occurred_at (RFC3339 UTC+Z), embedding
```
