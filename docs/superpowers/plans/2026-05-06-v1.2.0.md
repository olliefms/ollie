# Ollie v1.2.0 — Implementation Plan

**Branch:** `v1.2.0` (target: merge to `main` after ultrareview passes)  
**Date:** 2026-05-06  
**Scope:** Trips, drivers, trucks, trailers, journal/events, stop schema extension, blob query endpoint, and v1.1.1 bug fixes.

---

## Background

v1.1.0 shipped a RAG-enabled blob store, load/facility management with geocoding and routing, deduplication, semantic search, and a full OpenAPI spec. v1.2.0 adds the operational layer: the equipment and people that actually run the loads, the trips that represent physical movements, and the event infrastructure to support billing, audit, and future notifications.

---

## Guiding Principles

1. **LanceDB stays.** All new tables go into LanceDB. The future scaling path is pgvector (Postgres + vector extension), not SQLite. Do not introduce a second database engine in v1.2.
2. **Journal is additive.** Entity tables hold current state. The journal records history. Do not remove status fields from entities in favor of deriving them from the journal.
3. **Soft deletes everywhere.** Drivers, trucks, and trailers must be retained for historical trip records. `DELETE` endpoints set `status = inactive`, never remove rows.
4. **Status transitions are driven by events, not PUT.** `PUT` endpoints on drivers/trucks/trailers must reject attempts to manually set `assigned` or `dispatched` status. Those transitions happen only via trip action endpoints.
5. **Event hooks are stubs in v1.2.** Every action endpoint calls a hook in `src/events/mod.rs`. Hooks log only for now. v1.3 wires real side effects (email, push, EDI) without touching action logic.
6. **utoipa on everything.** Every new endpoint requires `#[utoipa::path(...)]`, schema derivations, and registration in `ApiDoc`. `llms.txt` must be updated when new resource groups are added. See AGENTS.md for the full checklist.

### Extract Cache

Extracted text is cached to disk at `{DATA_DIR}/extracts/ab/cd/<fullhash>.json`, parallel to blob storage. Uses the existing `shard_path()` logic. The pipeline writes the cache entry after `extract_content()` succeeds. The blob query endpoint reads it; on cache miss, re-extracts from stored blob bytes and writes the cache lazily.

Extract JSON shape:
```json
{ "type": "text", "content": "..." }
{ "type": "scanned_pdf", "content": "...", "raw_text": "..." }
```

### Trip Stop Shape

Trip stops are a JSON array on the trip record. They are richer than load stops — they include all stops a driver makes, not just freight pickups and deliveries. The full stop shape is defined in issue #27 (stop schema extension) and inherited by trip stops.

```json
{
  "sequence": 0,
  "stop_type": "origin | fuel | pickup | delivery | relay | empty_move | maintenance | terminal",
  "facility_id": "uuid or null",
  "name": "optional freeform name",
  "load_stop_index": null,
  "scheduled_arrive": "2026-05-10T08:00:00Z",
  "scheduled_arrive_end": null,
  "actual_arrive": null,
  "actual_depart": null,
  "expected_dwell_minutes": null,
  "detention_free_minutes": null,
  "detention_grace_minutes": null,
  "notes": null
}
```

When `actual_arrive` or `actual_depart` is written to a trip stop with a `load_stop_index`, cascade that value to `load.stops[load_stop_index]`.

### Detention Logic (Derived, Not Stored)

Computed at invoice time from stop fields:

- **FCFS** (`scheduled_arrive_end` is set): clock starts at `actual_arrive`
- **Strict** (`scheduled_arrive_end` is null):
  - If `actual_arrive − scheduled_arrive ≤ detention_grace_minutes` (default 15): clock starts at `scheduled_arrive`
  - If late beyond grace period: driver is **ineligible** for detention
- Detention = `max(0, actual_depart − clock_start − detention_free_minutes)` (default free time: 120 min)

### Load ↔ Trip Status Cascade

| Trip event | Load effect |
|-----------|-------------|
| First trip dispatched | Load → `dispatched` |
| First pickup stop departed | Load → `in_transit` |
| All trips delivered | Load → `delivered` |
| Last/only trip cancelled | Load → `planned` |

### Journal Table

LanceDB table named `events`. Append-only. BTree scalar indices on `entity_id`, `occurred_at`, `event_type`.

Internal write API:
```rust
pub async fn append_event(
    &self,
    entity_type: &str,
    entity_id: Uuid,
    event_type: &str,
    payload: Option<serde_json::Value>,
    actor: Option<&str>,
    occurred_at: &str,
) -> Result<(), AppError>
```

Events are read-only via the API. No `POST /events`.
