## Architecture decisions

### Why `assign_trip` never wrote driver/truck/trailer to the trip record

`assign_trip` was written to handle two concerns separately: (1) transition the trip's state machine and (2) update resource availability. The trip's `driver_id`/`truck_id`/`trailer_ids` fields were assumed to be set at trip creation time. In practice, hermes and the admin API create trips without pre-assigning resources, then call `/assign` ÔÇö leaving those fields null forever.

### Fix design

Add `DbClient::update_trip_resources(id, driver_id, truck_id, trailer_ids)` in `trip_ops.rs`. Follows the existing `upsert_trip` pattern: fetch Ôćĺ mutate Ôćĺ merge_insert. No schema changes ÔÇö `driver_id`, `truck_id`, and `trailer_ids` already exist on the trip record.

Call sites:
- `assign_trip`: set driver/truck/trailers on the trip after status transition (must not fail ÔÇö use `?` not `let _ =`)
- `unassign_trip`: clear to None/[] after status transition (same ÔÇö use `?`)

`complete_trip` must NOT clear these fields. They are the permanent historical record of who ran the trip. The existing `update_driver_status(Available)` calls are correct and will work once `existing.driver_id` is non-null.

### Error handling in `complete_trip`

The existing resource-release calls in `complete_trip` use `let _ =` (swallow errors). This is intentional and correct for completion ÔÇö a driver record lookup failure should not roll back a trip that physically completed. Leave as-is.

`assign_trip` and `unassign_trip` however must propagate errors from `update_trip_resources` with `?` ÔÇö a failed write here would leave the state machine inconsistent.

## Orchestrator brief

> **Rules for all subagents:**
> - Scope boundaries are hard. This is a backend-only fix. Do not touch `static/`.
> - Run `cargo build && cargo test && cargo clippy` before marking done.
> - Use `merge_insert` for all LanceDB updates. Never delete+insert.
> - `update_trip_resources` must use `?` propagation at the callsites in `assign_trip` and `unassign_trip` ÔÇö these are not fire-and-forget.
> - Do NOT modify `complete_trip` to clear driver/truck/trailer fields. Those must persist for history.
> - Update `#[utoipa::path]` annotation for `/assign` and `/unassign` if the response body changes (it shouldn't ÔÇö they return the trip record which already has these fields).
> - llms.txt does not need updating ÔÇö the endpoint behavior is unchanged from the documented contract.
