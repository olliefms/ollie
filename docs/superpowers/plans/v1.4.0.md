# v1.4.0 Release Plan — Dispatcher + Driver PWA Polish

**Branch:** `v1.4.0` (from `main` @ `f644054`)
**Model:** Two parallel execution tracks, one release branch, one PM merge.

---

## Overview

v1.4.0 ships two independent feature sets simultaneously:

| Track | Scope | PR target | Orchestrator |
|-------|-------|-----------|--------------|
| **Driver PWA** | PWA polish, stop-timeline fixes, URL state, passkey UX | `v1.4.0` | Session A |
| **Dispatcher** | Full dispatcher portal: models, auth, REST API, SPA dashboard, MCP server | `v1.4.0` | Session B |

Both sessions run in the **cloud environment** (ergofobe/ollie on GitHub). No local laptop required. Sessions end by opening a PR against `v1.4.0` and producing a hand-back prompt for the PM.

After both PRs are merged into `v1.4.0`, PM runs the standard release steps: ff-only merge to main, tag, GitHub release.

---

## Architecture decisions (pre-resolved)

**MCP library for #93:** Use `rmcp = "~1.5"` (crates.io stable). Mount `StreamableHttpService` via `.nest_service("/dispatch/mcp", service)`. Auth via `require_dispatcher_jwt` Axum middleware *before* the service — rmcp does not carry auth context into tool handlers. See issue #93 for full caveats.

**Dispatcher provisioning:** Dispatcher accounts are created by admin API only (no self-registration). Email uniqueness enforced at application layer (check-before-insert, 409 on collision).

**Design system:** Both apps follow `docs/DESIGN.md` (Linear-inspired, light theme, blue accent, surface-ladder hierarchy). Dispatcher SPA tokens live in `static/dispatch/css/base.css` and must import the same token set as Driver.

### MCP implementation notes (rmcp decision)

- Crate: `rmcp = "~1.5"` (crates.io). Do NOT float the version.
- Mount via `.nest_service("/dispatch/mcp", StreamableHttpService::new(...))` inside the Axum router, **after** `require_dispatcher_jwt` middleware is applied.
- Stateless operation only — no session store. Verify in smoke test that POSTs without `Mcp-Session-Id` succeed.
- Tool handlers are thin dispatch shims into existing DB ops. Business logic lives in the DB layer, not in tool handlers.
- Add a brief comment at the top of `src/api/dispatcher_portal/mcp.rs` documenting the rmcp choice and why.
- SSE stream endpoint (`GET /dispatch/mcp`) is optional — only implement if rmcp provides it for free. Stateless POST is sufficient for v1.4.

### Dispatcher test setup

Integration tests that exercise dispatcher endpoints need `DISPATCHER_JWT_SECRET` set. Mirror the `ADMIN_API_KEY=test-key` pattern: set `DISPATCHER_JWT_SECRET` to a 32-byte test string in the test helper. Example: `"test-dispatcher-secret-must-be-32b"`.

## PM release steps (after both hand-backs received)

Standard patch/minor release flow from `feedback_orchestrator_workflow.md`:

1. Review both PRs — spot-check changed files, confirm Opus PASS on both
2. Merge Driver PWA PR into `v1.4.0` (`gh pr merge <N> --merge`)
3. Merge Dispatcher PR into `v1.4.0` (`gh pr merge <N> --merge`)
4. Verify `cargo build && cargo test && cargo clippy` on `v1.4.0` with both merged
5. Run a final Opus review of the combined `v1.4.0` branch (subagent) — focus on interactions between tracks if any
6. Update `AGENTS.md` with v1.4.0 sprint lessons; commit to `v1.4.0`
7. Validate and close all in-scope issues with verification comments
8. `git checkout main && git merge --ff-only v1.4.0 && git push origin main`
9. Confirm tips match: `git log --oneline -1 main && git log --oneline -1 v1.4.0`
10. `git tag v1.4.0 && git push origin refs/tags/v1.4.0`
11. `gh release create v1.4.0 --title "..." --notes "..." --target main`

**New production env var required:** `DISPATCHER_JWT_SECRET` (min 32 bytes, arbitrary string). Must be set on the production server before deploying v1.4.0. Flag this in the GitHub release notes.
