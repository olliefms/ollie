## Overview

v1.3.0 ships a mobile-first PWA driver portal at `/driver`. Drivers log in with a passkey (WebAuthn) or PIN, then view their assigned trips — past, current, and upcoming — with full stop detail. Read-only for this version; driver actions deferred to v1.4+.

Also includes three open bug/feature issues carried forward from v1.2: #21 (facility dedup error swallowing), #22 (search count rename), #25 (blob query endpoint).

**Design spec:** `docs/superpowers/specs/2026-05-07-driver-ui-design.md`

## Architecture Review Decisions

Opus review flagged the following — all resolved inline:

| Finding | Risk | Resolution |
|---------|------|------------|
| WebAuthn counter race (JSON blob) | Medium | Separate `driver_passkey_credentials` table keyed on `credential_id`; counter updated per-auth via merge_insert |
| JWT secret management | Medium | Dedicated `DRIVER_JWT_SECRET` env var (≥32 bytes); panic at startup if missing. Claims: `driver_id`, `token_version`, `iss`, `aud`, `exp`, `iat`, `kid` |
| Tab classification (status only) | Medium | Time + status logic: upcoming = planned/assigned AND first stop in future; current = dispatched/in_transit OR assigned AND first stop now/past |
| Sequential LanceDB queries | Medium | `tokio::try_join!` batches facility lookup + load fetch concurrently on trips/:id |
| WebAuthn challenge state in LanceDB | High | In-process `Arc<DashMap>` with 5-min TTL + background sweep; single-instance only for v1.3 |
| ServeDir vs API mount order | Low | API router nested first; ServeDir with index.html fallback mounted second |
| RP ID / localhost | Low | `DRIVER_RP_ID` and `DRIVER_RP_ORIGIN` env vars; separate dev/prod builder instances |
| Phone timing attack | Low | Return same response shape for unknown phone; rate-limit `/auth/challenge` at 10/min per IP |
| PIN brute-force | High | 5-attempt lockout with exponential backoff; `failed_pin_attempts` + `locked_until` in `driver_credentials`; IP rate-limit on `/auth/pin` |
| JWT revocation | Medium | `token_version` in `driver_credentials` + JWT claim; bumped on PIN reset, passkey removal, status → inactive; checked on every authenticated request |
| Inactive driver not blocked | High | Status check at `/auth/verify` and `/auth/pin`; token_version mismatch = 401 in middleware |
| Concurrent passkey registration | Medium | Fixed by separate passkey table |
| No audit log | Medium | Structured tracing log lines for all auth events (no new LanceDB table in v1.3) |
| Refresh endpoint scope creep | Medium | Refresh refused if JWT `iat` > 7 days old |
| Phone normalization | Low | E.164 normalize on `PUT /api/v1/drivers/:id` write and on phone lookup during auth |
| WebAuthn user handle | Medium | `driver_id` UUID used as `user.id` — not phone number |
| SW intercepts API routes | Low | SW explicitly excludes `/driver/api/v1/*` from cache |

## Non-Negotiable Architecture Rules (unchanged from v1.2)

- `merge_insert` everywhere — upserts only, never delete+insert
- RFC3339 UTC+Z on all timestamps
- LanceDB stays — no SQLite, no new relational DB
- Soft deletes — status = inactive, never hard delete
- Status transitions via events only (trip assign/dispatch endpoints)

---

## Out of Scope for v1.3

- Driver actions (arrive/depart, check-calls, document uploads)
- Push notifications
- Bottom nav bar (hidden in DOM, not rendered)
- Structured reference number fields on loads
- Offline trip data caching (service worker caches assets only)
- Horizontal scaling / Redis for challenge state
- Second ultrareview pass (unless structural changes made during fix wave)
